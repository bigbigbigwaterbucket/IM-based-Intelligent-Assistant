# 文档生成 Skill

## 通用规则

- 文档必须面向最终用户，不描述 agent 自身的思考、规划、工具调用或中间状态。
- 不要出现“意图分析”“执行计划”“工具调用”“规划来源”“agent 产物”等内部过程标题。
- 优先使用已获取的聊天上下文、原始消息、会议纪要或用户提供材料；不要编造事实、结论、人员、日期或指标。
- 如果上下文不足，明确写“待补充信息”，并说明缺少哪类信息。
- 内容要可核查：保留关键原文摘录、消息来源或能让用户回查的线索。
- 标题和章节使用简洁中文；不需要为了填充篇幅制造空段落。

## 模板选择

根据用户指令和上下文选择一个最匹配的模板。用户明确要求某类文档时，以用户要求为准；没有明确类型时，聊天消息整理默认使用“聊天总结文档”。

## 模板 A：聊天总结文档

适用场景：用户要求“总结聊天消息”“整理群聊”“生成聊天记录文档”。

建议结构：

1. 标题
2. 摘要
3. 关键讨论点
4. 已形成结论
5. 待办事项
6. 风险与待确认问题
7. 原始消息摘录

写作要求：

- 摘要只总结真实消息中出现的信息。
- 待办事项包含负责人和时间要求；如果消息里没有，标记“待补充信息”。
- 原始消息摘录保留能支撑结论的代表性内容，不要把无关系统日志放进去。

## 模板 B：需求文档

适用场景：用户要求“需求文档”“PRD”“产品需求”“功能需求整理”。

建议结构：

1. 背景与目标
2. 用户/角色
3. 需求范围
4. 功能需求
5. 非功能需求
6. 交互与流程
7. 验收标准
8. 风险与依赖
9. 待补充信息

写作要求：

- 功能需求使用可验证描述，避免泛泛而谈。
- 验收标准尽量写成可测试条目。
- 对聊天中没有明确的信息，不要自行扩展为确定需求。

## 模板 C：产品文档

适用场景：用户要求“产品文档”“功能说明”“使用说明”“对外介绍文档”。

建议结构：

1. 产品/功能概述
2. 使用对象
3. 核心能力
4. 使用流程
5. 配置项或限制
6. 常见问题
7. 更新记录或后续计划

写作要求：

- 面向读者说明产品或功能怎么用、解决什么问题。
- 避免暴露实现细节，除非用户明确要求技术文档。
- 限制和前提条件要写清楚。

## 模板 D：方案/项目汇报文档

适用场景：用户要求“方案文档”“项目汇报”“进展汇总”“给领导看的文档”。

建议结构：

1. 背景
2. 目标
3. 当前进展
4. 关键方案
5. 结果或预期收益
6. 风险与应对
7. 下一步计划

写作要求：

- 先讲结论，再展开依据。
- 进展、收益、风险必须来自上下文；没有数据时标记“待补充信息”。

## 模板 E：会议纪要/行动项文档

适用场景：聊天内容是会议讨论、纪要、同步会，或用户要求“会议纪要”“行动项”。

建议结构：

1. 会议主题
2. 参会人
3. 讨论摘要
4. 决议
5. 行动项
6. 待确认问题
7. 原文依据

写作要求：

- 行动项尽量包含事项、负责人、截止时间、状态。
- 未提及的参会人或时间不要编造。
